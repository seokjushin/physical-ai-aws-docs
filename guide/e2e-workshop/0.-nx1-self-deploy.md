# 0. NX1 사전 준비 (먼저 읽기)

이 워크샵은 **Doosan Bobcat BEST NX1** 계정에서 진행됩니다. NX1은 일반 AWS 계정과 거버넌스가 달라, 원본 가이드의 **CloudShell · `cdk deploy` · SSH · DCV 공인 URL · CloudFront** 방식이 모두 막혀 있습니다.

(단 `BESTNX1-Developer-737138011740` 프로파일 이름과 `aws sts get-caller-identity` 출력 예시의 계정 번호는 prescribed 값이라 그대로 사용합니다.)

## ⚠️ NX1 핵심 차이 (4줄 요약)

- **리소스 직접 생성 불가** → 배포는 **CloudFormation 콘솔** (서비스롤 `CloudFormationDeployer` + 태그 `doosan:owner=BEST_NX1` 강제)
- **CloudShell 차단, SSH 차단** → CLI는 **SSM Session Manager**, GUI(DCV/code-server)는 **SSM 포트포워딩**
- **공유 VPC** (per-user VPC 생성 불가) → 본인 `UserId`로만 리소스 구분. 다른 사람과 겹치지 않는 ID(소문자·숫자·하이픈)
- **한 번에 완료** — 인스턴스는 무활동 시 자동 정지되고 **본인은 다시 켤 수 없습니다** (강사만). 자리를 오래 비우지 마세요.

> 🔴 **2026-05-28 이전에 본인 Day1 스택을 만든 분 (재배포 필요)**: 초기 템플릿이 SSM 세션 S3 로그 권한을 누락해 Session Manager 시작 시 `AccessDenied: s3:GetEncryptionConfiguration on ... db-bestnx1-us-east-1-sessionmanager-logs` 에러로 종료됩니다. 콘솔 → CloudFormation → 본인 스택 → **Update stack** → *Replace current template* → **Amazon S3 URL** 입력: `<DAY1_TEMPLATE_URL>` (2026-05-28 16:59 UTC 재업로드 완료) → 그대로 진행. UserData 재실행 없이 IAM 정책만 갱신되어 ~1분 내 `UPDATE_COMPLETE`. 또는 Delete 후 재배포.

## 사전 준비 (로컬 PC, 1회)

다음 4단계는 워크샵 시작 전에 한 번만 끝내면 이후 모든 모듈에서 재사용됩니다.

### 1. AWS Identity Center 로그인 확인

- 포털 URL: `https://ad-doosanad-1s129xqo6hgkh.awsapps.com/start/` (강사 제공 또는 북마크)
- Doosan AD 계정 + MFA로 로그인
- 콘솔에서 **AWS account → BEST NX1 → BESTNX1-Developer** 역할이 보이는지 확인

### 2. AWS CLI v2 + Session Manager plugin 설치

DCV/code-server GUI 접속에 SSM 포트포워딩이 필요하므로 둘 다 필수입니다.

- AWS CLI v2: https://aws.amazon.com/cli/
- Session Manager plugin: https://docs.aws.amazon.com/systems-manager/latest/userguide/session-manager-working-with-install-plugin.html

설치 후 확인:

```bash
aws --version              # aws-cli/2.x.x  (v1이면 안 됨)
session-manager-plugin     # The Session Manager plugin was installed successfully...
```

### 3. `~/.aws/config` 작성 (1회)

`aws sso login` 실행 전에 반드시 이 블록이 `~/.aws/config`에 있어야 합니다. 없으면 `The config profile (BESTNX1-Developer-737138011740) could not be found` 에러가 납니다.

- macOS/Linux: `~/.aws/config`
- Windows: `%USERPROFILE%\.aws\config` (예: `C:\Users\<본인>\.aws\config`)

```ini
[profile BESTNX1-Developer-737138011740]
sso_session = nx1
sso_account_id = 737138011740
sso_role_name = BESTNX1-Developer
region = us-east-1

[sso-session nx1]
sso_start_url = https://ad-doosanad-1s129xqo6hgkh.awsapps.com/start/
sso_region = us-east-1
sso_registration_scopes = sso:account:access
```

> Profile/sso-session 이름은 위 그대로 사용하세요 (가이드와 명령들이 모두 이 이름을 가정). DoosanICA `db-githubactions-aws` 리포의 prescriptive AWS config 가이드와도 일치합니다.
>
> **`aws configure sso` 위저드는 권장하지 않음** — 위저드는 마지막에 profile 이름을 묻고, 기본값이 prescribed와 달라 한 글자 실수로 후속 명령들이 깨집니다 (Doosan 측 canned actions·runbook이 모두 정확한 이름 가정). 위 블록을 그대로 복붙하는 게 실패율이 낮습니다. 이미 위저드로 다른 이름으로 만들었다면 해당 블록만 위 형태로 교체하세요.

### 4. 하루 1회 SSO 로그인

```bash
aws sso login --profile BESTNX1-Developer-737138011740
```

브라우저가 열리고 Doosan AD 로그인 → 토큰 8~24시간 유효. 다음으로 동작 확인:

```bash
aws sts get-caller-identity --profile BESTNX1-Developer-737138011740
# Account: 737138011740, Arn: ...assumed-role/AWSReservedSSO_BESTNX1-Developer_.../<본인>@doosan.com
```

## 자주 막히는 곳 (사전 준비 관련)

| 증상 | 해결 |
|---|---|
| `aws sso login` 시 `The config profile (BESTNX1-Developer-737138011740) could not be found` | 위 §3의 profile + sso-session 블록을 `~/.aws/config`에 추가 (Windows: `%USERPROFILE%\.aws\config`) |
| `aws sso login` 시 브라우저 안 열림 / `unsupported scheme` | AWS CLI v1입니다. v2로 업그레이드 (`aws --version`이 `aws-cli/2.x` 인지 확인) |
| 토큰 만료 (`The SSO session has expired`) | `aws sso login --profile BESTNX1-Developer-737138011740` 다시 실행. 토큰은 8~24시간 |
| `session-manager-plugin: command not found` | Session Manager plugin 미설치. 위 §2 링크에서 OS별 설치 |
| Identity Center 포털에 BEST NX1 계정이 안 보임 | AD 그룹 할당이 누락됐을 수 있음. 강사 또는 Dimitri에게 SR 신청 |

> 인프라 배포·SSM 세션·DCV·MLflow 관련 트러블슈팅은 각 모듈의 "자주 막히는 곳" 표를 참고하세요.
