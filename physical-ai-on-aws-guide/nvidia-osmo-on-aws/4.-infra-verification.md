# 4. 인프라 검증

배포 및 설정이 완료된 인프라가 실제로 동작하는지 테스트 Job을 실행하여 검증합니다. 이 단계를 건너뛰면 Workflow 실행 시 문제 원인을 파악하기 어려우므로, 반드시 수행합니다.

---

### 4.1 S3 연결 테스트

IRSA(IAM Roles for Service Accounts)를 통해 Pod에서 S3에 정상적으로 접근할 수 있는지 확인합니다. IRSA가 정상 동작하면 Pod는 별도의 AWS 자격 증명 없이 IAM Role의 권한으로 S3에 파일을 읽고 쓸 수 있습니다.

먼저 CDK가 생성한 S3 버킷 이름을 CloudFormation Stack Output에서 가져옵니다:

```bash
BUCKET=$(aws cloudformation describe-stacks --stack-name Osmo \
  --query 'Stacks[0].Outputs[?OutputKey==`S3BucketName`].OutputValue' --output text)

kubectl apply -f - <<EOF
apiVersion: batch/v1
kind: Job
metadata:
  name: test-s3
  namespace: osmo
spec:
  template:
    spec:
      serviceAccountName: osmo-workload
      nodeSelector:
        node-role: system
      containers:
      - name: s3-test
        image: amazon/aws-cli:latest
        command:
        - /bin/sh
        - -c
        - |
          echo "=== S3 IRSA Test ==="
          echo "hello-osmo-\$(date +%s)" > /tmp/test.txt
          aws s3 cp /tmp/test.txt s3://${BUCKET}/test/infra-test.txt
          aws s3 ls s3://${BUCKET}/test/
          echo "=== S3 TEST PASSED ==="
      restartPolicy: Never
  backoffLimit: 1
EOF
```

30초 후 결과 확인:

```bash
kubectl logs -n osmo -l job-name=test-s3
```

**정상 출력:**

```
=== S3 IRSA Test ===
upload: ../tmp/test.txt to s3://osmo-data-osmo-.../test/infra-test.txt
2026-05-03 10:25:47         22 infra-test.txt
=== S3 TEST PASSED ===
```

{% hint style="success" %}
`S3 TEST PASSED`가 출력되면 IRSA → S3 연결이 정상입니다.
{% endhint %}

### 4.2 Cluster Autoscaler 동작 검증

GPU 리소스를 요청하는 Pod를 생성하여 Cluster Autoscaler가 노드를 추가하는지 확인합니다. 현재 GPU 노드는 0대이므로, GPU를 요청하는 Pod가 Pending 상태가 되고, CA가 이를 감지하여 ASG에 scale-up을 요청하는 것이 정상 동작입니다.

```bash
kubectl apply -f - <<'EOF'
apiVersion: batch/v1
kind: Job
metadata:
  name: test-gpu-autoscale
  namespace: osmo
spec:
  template:
    spec:
      serviceAccountName: osmo-workload
      nodeSelector:
        node-role: gpu-sim
      tolerations:
      - key: nvidia.com/gpu
        operator: Exists
        effect: NoSchedule
      containers:
      - name: gpu-test
        image: nvidia/cuda:12.4.0-base-ubuntu22.04
        command: ["nvidia-smi"]
        resources:
          limits:
            nvidia.com/gpu: 1
      restartPolicy: Never
  backoffLimit: 1
EOF
```

Pod 상태 확인:

```bash
kubectl get pods -n osmo -l job-name=test-gpu-autoscale
```

**예상:** `Pending` 상태 (GPU 노드가 아직 없으므로)

Cluster Autoscaler 로그 확인:

```bash
kubectl logs -n kube-system -l app.kubernetes.io/name=aws-cluster-autoscaler | \
  grep -E "(scale-up|Scale-up|unschedulable)"
```

**정상 출력:**

```
Pod osmo/test-gpu-autoscale-xxxxx is unschedulable
Scale-up: setting group eks-gpu-sim-... size to 1 instead of 0 (max: 8)
```

{% hint style="info" %}
Cluster Autoscaler가 `Scale-up` 메시지를 출력하면 오토스케일링이 정상 동작하는 것입니다. 실제 GPU 인스턴스 프로비저닝은 AWS 리전의 capacity에 따라 2~5분 소요됩니다.

GPU 인스턴스 용량이 부족한 경우(`InsufficientInstanceCapacity`) AWS 리전의 재고 문제이며, 인프라 설정의 문제가 아닙니다.
{% endhint %}

### 4.3 ASG 상태 직접 확인

AWS CLI로 Auto Scaling Group의 상태를 직접 확인합니다. CA가 내부적으로 모니터링하는 ASG와 실제 AWS 상태가 일치하는지 검증하는 단계입니다:

```bash
# 모든 노드그룹 ASG 확인
aws autoscaling describe-auto-scaling-groups \
  --query 'AutoScalingGroups[?contains(AutoScalingGroupName, `eks-`)].{Name:AutoScalingGroupName,Desired:DesiredCapacity,Min:MinSize,Max:MaxSize}' \
  --output table
```

**정상 출력:**

```
-------------------------------------------------------------------
|                  DescribeAutoScalingGroups                        |
+------------------+----------+------+------+
|      Name        | Desired  | Min  | Max  |
+------------------+----------+------+------+
| eks-system-...   |    2     |  2   |  3   |
| eks-gpu-sim-...  |    1     |  0   |  8   |
| eks-gpu-train-...|    0     |  0   |  4   |
+------------------+----------+------+------+
```

### 4.4 GPU 노드 접속 후 nvidia-smi 확인

GPU 노드가 Ready 상태가 되면 테스트 Job의 로그를 확인합니다:

```bash
# GPU 노드 상태 확인
kubectl get nodes -l node-role=gpu-sim

# Job 로그 확인
kubectl logs -n osmo -l job-name=test-gpu-autoscale
```

**정상 출력 (nvidia-smi):**

```
+-----------------------------------------------------------------------------------------+
| NVIDIA-SMI 550.x        Driver Version: 550.x        CUDA Version: 12.4               |
|-------------------------------------------+----------------------+---------------------+
| GPU  Name                 Persistence-M   | Bus-Id        Disp.A | Volatile Uncorr. ECC |
|   0  NVIDIA L4                      On    | 00000000:00:1E.0 Off |                    0 |
+-------------------------------------------+----------------------+---------------------+
```

### 4.5 테스트 리소스 정리

검증이 완료되면 테스트 Job을 삭제합니다:

```bash
kubectl delete job test-s3 test-gpu-autoscale -n osmo

# GPU 노드를 다시 0으로 축소 (비용 절감)
# Cluster Autoscaler가 10분 idle 후 자동으로 축소하지만, 수동으로도 가능:
aws autoscaling describe-auto-scaling-groups \
  --query 'AutoScalingGroups[?contains(AutoScalingGroupName, `gpu-sim`)].AutoScalingGroupName' \
  --output text | xargs -I{} aws autoscaling set-desired-capacity \
  --auto-scaling-group-name {} --desired-capacity 0
```

### 검증 요약

| 항목 | 검증 방법 | 성공 기준 |
|------|-----------|-----------|
| kubectl 접근 | `kubectl get nodes` | 2개 노드 Ready |
| S3 IRSA | test-s3 Job | `S3 TEST PASSED` 출력 |
| Cluster Autoscaler | test-gpu Job + CA 로그 | `Scale-up` 메시지 확인 |
| GPU 노드 | `nvidia-smi` in Pod | GPU 정보 출력 |
