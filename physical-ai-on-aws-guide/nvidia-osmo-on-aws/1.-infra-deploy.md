# 1. 인프라 배포 (CDK)

AWS CDK를 사용하여 NVIDIA OSMO를 위한 EKS 기반 인프라를 자동으로 프로비저닝합니다. VPC, EKS 클러스터, GPU 노드그룹, RDS PostgreSQL, ElastiCache Redis, S3 등 모든 리소스가 한 번의 명령으로 생성됩니다.

### 사전 요구 사항

* AWS 계정 — 권장 리전: `us-east-1` (GPU 인스턴스 capacity 가용성이 높음)
* GPU 인스턴스 서비스 할당량 확인 — AWS 콘솔 → [Service Quotas](https://console.aws.amazon.com/servicequotas/) → EC2 → `Running On-Demand G and VT instances` 값이 필요한 vCPU 수 이상인지 확인
  * g5.12xlarge = 48 vCPU (Sim 노드)
  * g6e.12xlarge = 48 vCPU (Train 노드)

{% hint style="warning" %}
GPU 인스턴스 할당량이 부족하면 노드가 프로비저닝되지 않습니다. 배포 전 Service Quotas에서 할당량을 확인하세요.
{% endhint %}

### 1.1 CloudShell 환경 설정 (~5분)

1. [AWS 콘솔](https://console.aws.amazon.com)에 로그인합니다.
2. 우측 상단 리전을 **US East (N. Virginia) `us-east-1`** 로 설정합니다.
3. 상단 `>_` 아이콘을 클릭하여 CloudShell을 실행합니다.
4. 아래 명령어를 실행합니다:

```bash
# 리포지토리 클론
git clone --depth 1 https://github.com/hi-space/aws-physical-ai-recipes.git ~/aws-physical-ai-recipes

# CDK 프로젝트 설정
cd ~/aws-physical-ai-recipes/osmo/cdk
npm install

# CDK CLI 확인
npx cdk --version

# kubectl 설치
curl -LO "https://dl.k8s.io/release/v1.30.0/bin/linux/amd64/kubectl"
chmod +x kubectl && sudo mv kubectl /usr/local/bin/

# Helm 설치
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
```

{% hint style="info" %}
CloudShell 재접속 시에는 `cd ~/aws-physical-ai-recipes/osmo/cdk`만 실행하면 됩니다.
{% endhint %}

---

### 1.2 CDK Bootstrap (최초 1회)

해당 리전에서 CDK를 처음 사용하는 경우 bootstrap을 실행합니다:

```bash
npx cdk bootstrap aws://$(aws sts get-caller-identity --query Account --output text)/us-east-1
```

---

### 1.3 인프라 배포 (~20분)

```bash
npx cdk deploy --require-approval never
```

{% hint style="info" %}
CloudShell 세션 타임아웃이 걱정된다면 백그라운드로 실행합니다:

```bash
nohup npx cdk deploy --require-approval never > deploy.log 2>&1 &
tail -f deploy.log
```
{% endhint %}

### 1.4 배포 완료 확인

배포가 완료되면 CloudFormation Output이 출력됩니다:

```bash
aws cloudformation describe-stacks --stack-name Osmo \
  --query 'Stacks[0].Outputs[*].{Key:OutputKey,Value:OutputValue}' \
  --output table
```

**출력 예시:**

| Key | Value |
|-----|-------|
| EksClusterName | osmo-eks |
| S3BucketName | osmo-data-osmo-123456789012-us-east-1 |
| VpcId | vpc-0xxxxxxxxxxxxxxxxx |
| OsmoDbEndpoint | osmo-postgres.xxx.us-east-1.rds.amazonaws.com |
| OsmoDbPort | 5432 |
| OsmoRedisEndpoint | osmo-redis.xxx.cache.amazonaws.com |
| OsmoRedisPort | 6379 |
| OsmoDbSecretArn | arn:aws:secretsmanager:us-east-1:... |

{% hint style="success" %}
`EksClusterName`이 출력되면 배포가 성공한 것입니다. 다음 단계로 진행하세요.
{% endhint %}

### 1.5 멀티유저 배포 (워크숍용)

여러 참가자가 동시에 실습하는 워크숍에서는 `userId` 파라미터로 격리된 환경을 생성합니다:

```bash
npx cdk deploy \
  -c userId=alice \
  -c vpcCidr=10.1.0.0/16 \
  --require-approval never
```

<details>

<summary>배포 파라미터 요약</summary>

| 파라미터 | 설명 | 기본값 | 예시 |
|----------|------|--------|------|
| `userId` | 참가자 식별자 | (없음) | `alice` |
| `vpcCidr` | VPC CIDR (참가자별 고유) | `10.0.0.0/16` | `10.1.0.0/16` |
| `gpuSimMaxNodes` | Sim 노드 최대 수 | `8` | `4` |
| `gpuTrainMaxNodes` | Train 노드 최대 수 | `4` | `2` |

</details>

### Troubleshooting

| 증상 | 원인 | 해결 |
|------|------|------|
| `osmo-db-secret already exists` | 이전 배포의 잔여 Secret | `aws secretsmanager delete-secret --secret-id osmo-db-secret --force-delete-without-recovery` |
| ROLLBACK_COMPLETE 상태 | 리소스 생성 실패 후 롤백 | `aws cloudformation delete-stack --stack-name Osmo` 후 재배포 |
| 배포 20분 이상 대기 | RDS 생성에 시간 소요 | 정상. CloudFormation 콘솔에서 진행 상황 확인 |
