# 4. VLA 학습 실행 (GR00T Fine-tuning)

## VLA(Vision-Language-Action)란?

**VLA(Vision-Language-Action) 모델**은 시각 입력(카메라 이미지)과 언어 지시사항을 이해하고, 로봇의 행동(움직임)을 생성하는 멀티모달 기초 모델입니다. VLA는 기존의 이미지 분류나 객체 인식 모델과 달리, 로봇이 실제로 수행해야 할 **구체적인 작업**(로봇 팔의 위치, 속도, 힘 등)을 출력합니다.

### GR00T N1.7의 특징

| 항목 | 내용 |
|------|------|
| 모델 크기 | 3B 파라미터 |
| Backbone | Cosmos-Reason2-2B (Qwen3-VL 아키텍처) |
| Action 표현 | Relative EEF (현재 자세 대비 상대 변위) |
| 사전학습 데이터 | 20K 시간 EgoScale 인간 영상 + 로봇 데모 |
| Fine-tuning 방식 | Projector + Diffusion Model 선택적 학습 |
| 지원 데이터셋 | LeRobot v2 포맷 |

### Fine-tuning이 중요한 이유

사전학습된 GR00T 모델은 일반적인 로봇 작업에 잘 작동하지만, 특정 로봇 플랫폼(예: ALOHA, Mobile Manipulator)과 특정 작업(예: 서랍 열기, 물건 집기)에 대해서는 **커스텀 데이터셋으로 추가 학습**(Fine-tuning)이 필요합니다. 이렇게 하면:

1. **작업 특화 성능**: 모델이 특정 작업에 더 정확한 행동을 생성
2. **데이터 효율성**: 적은 데이터량(수백~수천 샘플)으로 빠르게 학습 가능
3. **비용 효율성**: GPU 1개에서도 학습 가능하여 인프라 비용 절감

---

SLURM을 사용하여 GR00T VLA 모델의 Fine-tuning 작업을 제출합니다. HyperPod가 자동으로 GPU 노드를 프로비저닝하고, 학습이 끝나면 노드를 해제합니다.

---

## 4.1 학습 환경 준비

앞서 **2.6 워크숍 환경 초기화**에서 실행한 `setup_environment.sh` 스크립트가 이미 다음을 준비했습니다:

- GR00T N1.7 컨테이너 이미지 (`gr00t-core:1.7.0`) 다운로드
- Enroot를 통한 컨테이너 이미지 변환 (`nvidia+gr00t+gr00t-core+1.7.0.sqsh`)
- 필요한 Python 패키지 설치

필요시 컨테이너 이미지를 수동으로 재가져올 수 있습니다:

```bash
# Enroot로 GR00T N1.7 컨테이너 이미지 재가져오기 (선택사항)
cd /fsx/scratch
enroot import docker://nvcr.io/nvidia/gr00t/gr00t-core:1.7.0
```

{% hint style="info" %}
**Enroot란?** HyperPod에서 컨테이너를 실행하기 위한 도구입니다. Docker와 달리 root 권한 없이 SLURM 작업 내에서 컨테이너를 실행할 수 있습니다.
{% endhint %}

## 4.2 학습 스크립트

학습 스크립트는 레포지토리에 포함되어 있습니다:

```bash
# 학습 스크립트 확인
cat /fsx/scratch/aws-physical-ai-recipes/training/hyperpod/examples/vla/train_groot.py
```

주요 기능:
- `--dataset-path`: LeRobot v2 형식 데이터셋 경로
- `--embodiment-tag`: 로봇 플랫폼 태그
- `--max-steps`, `--batch-size`, `--learning-rate`: 학습 하이퍼파라미터
- MLflow 자동 통합 (MLFLOW_TRACKING_URI 환경변수 설정 시)

### 실제 GR00T N1.7 CLI 명령

GR00T N1.7은 다음과 같은 CLI를 지원합니다:

```bash
# 기본 fine-tuning 명령 (단일 GPU)
python -m gr00t.core.scripts.train \
  --dataset-path /fsx/datasets/groot/aloha \
  --embodiment-tag new_embodiment \
  --num-gpus 1 \
  --batch-size 32 \
  --max-steps 5000 \
  --learning-rate 2e-5 \
  --output-dir /fsx/checkpoints/vla/groot-n1.7-aloha

# 멀티 GPU fine-tuning (분산 학습)
torchrun --nproc_per_node=4 -m gr00t.core.scripts.train \
  --dataset-path /fsx/datasets/groot/aloha \
  --embodiment-tag new_embodiment \
  --num-gpus 4 \
  --batch-size 128 \
  --max-steps 10000 \
  --output-dir /fsx/checkpoints/vla/groot-n1.7-aloha
```

**주요 파라미터:**

| 파라미터 | 설명 | 기본값 |
|---------|------|--------|
| `--dataset-path` | LeRobot v2 포맷 데이터셋 경로 | 필수 |
| `--embodiment-tag` | 로봇 플랫폼 태그 (예: aloha, franka) | 필수 |
| `--num-gpus` | 사용할 GPU 수 | 1 |
| `--batch-size` | 배치 크기 | 32 |
| `--max-steps` | 최대 학습 스텝 | 10000 |
| `--learning-rate` | 학습률 | 2e-5 |
| `--output-dir` | 체크포인트 저장 경로 | ./outputs |
| `--data-config` | 카메라/로봇 구성 | default |

## 4.3 SLURM 작업 제출

```bash
# 기본 학습 작업 제출 (ALOHA 데이터셋, GPU 4개)
/fsx/scratch/aws-physical-ai-recipes/training/hyperpod/slurm-templates/vla/run_vla.sh --dataset aloha

# 또는 직접 sbatch 사용 (파라미터 커스터마이즈)
sbatch --export=ALL,DATASET=aloha,MAX_STEPS=10000,BATCH_SIZE=64 \
  /fsx/scratch/aws-physical-ai-recipes/training/hyperpod/slurm-templates/vla/finetune_groot.sbatch
```

작업 제출 시 출력:

```
Submitted batch job 12345
```

{% hint style="warning" %}
**HyperPod 노드 프로비저닝:** `--partition=train`으로 지정하면 HyperPod가 `ml.g6e.12xlarge` (L40S×4) 노드를 자동으로 프로비저닝합니다. 노드 시작까지 수분이 걸릴 수 있습니다. 초기에는 `PD(Pending)` 상태를 유지합니다.
{% endhint %}

## 4.4 SLURM 개념 설명

HyperPod에서 학습 작업을 효율적으로 관리하기 위해 알아야 할 SLURM 명령:

### sbatch: 배치 작업 제출

```bash
# 스크립트 파일로 작업 제출
sbatch train_job.sh

# 인라인으로 작업 제출
sbatch << 'EOF'
#!/bin/bash
#SBATCH --nodes=1
#SBATCH --gres=gpu:4
echo "Hello from SLURM"
EOF
```

**주요 옵션:**

| 옵션 | 설명 |
|------|------|
| `--job-name` | 작업 이름 |
| `--partition` | 작업 큐 (train, interactive 등) |
| `--nodes` | 요청 노드 수 |
| `--gres=gpu:N` | GPU 수 (N = 1, 2, 4, 8 등) |
| `--time` | 최대 실행 시간 (HH:MM:SS) |
| `--output` | 표준 출력 파일 |
| `--error` | 표준 에러 파일 |

### squeue: 작업 큐 확인

```bash
# 모든 작업 확인
squeue

# 특정 작업 상태 확인
squeue -j <JOB_ID>

# 현재 사용자의 작업만 확인
squeue -u $USER

# 상세 정보 출력
squeue --long

# 실시간 모니터링 (2초마다 새로고침)
watch squeue
```

출력 예:
```
             JOBID PARTITION     NAME     USER ST       TIME  NODES NODELIST(REASON)
             12345      train groot-n1 ubuntu    R    10:25      1 ip-10-0-5-42
             12346      train test-run ubuntu    PD       0:00      1 (Priority)
```

**상태 설명:**

| 상태 | 코드 | 의미 |
|------|------|------|
| Pending | PD | GPU 노드 프로비저닝 대기 중 |
| Running | R | 학습 실행 중 |
| Completed | CD | 정상 완료 |
| Failed | F | 오류로 종료 |
| Cancelled | CA | 사용자가 취소 |
| Configuring | CF | 노드 설정 중 |
| Completing | CG | 정리 중 |

### scontrol: 작업 상태 조회 및 수정

```bash
# 작업 상세 정보 조회
scontrol show job <JOB_ID>

# 작업 우선순위 변경
scontrol update JobId=<JOB_ID> Priority=10
```

### scancel: 작업 취소

```bash
# 특정 작업 취소
scancel <JOB_ID>

# 사용자의 모든 작업 취소
scancel -u $USER

# 특정 사용자의 모든 Pending 작업 취소
scancel -u <USERNAME> -t PENDING
```

## 4.5 작업 모니터링

### 실시간 로그 확인

```bash
# 작업 로그 실시간 확인
tail -f /fsx/scratch/logs/groot-<JOB_ID>.out

# 에러 로그 확인
tail -f /fsx/scratch/logs/groot-<JOB_ID>.err

# 마지막 100줄 확인
tail -100 /fsx/scratch/logs/groot-<JOB_ID>.out
```

### 작업 상태 모니터링

```bash
JOB_ID=12345  # 할당받은 Job ID로 변경

# 작업 상태 확인
squeue -j $JOB_ID

# 5초마다 자동 새로고침
watch -n 5 squeue -j $JOB_ID

# GPU 할당 확인
scontrol show job $JOB_ID | grep Exec
```

### GPU 메모리 모니터링 (학습 중)

작업이 Running 상태일 때:

```bash
# Head Node에서 GPU 노드에 SSH 접속 후 nvidia-smi 실행
# (또는 로그에서 GPU 메모리 사용량 확인)
tail -50 /fsx/scratch/logs/groot-<JOB_ID>.out | grep -i memory
```

## 4.6 멀티노드 분산 학습 (선택)

여러 노드에 걸쳐 분산 학습을 실행할 수 있습니다:

```bash
sbatch << 'EOF'
#!/bin/bash
#SBATCH --job-name=groot-n1.7-multinode
#SBATCH --partition=train
#SBATCH --nodes=2
#SBATCH --ntasks-per-node=4
#SBATCH --gres=gpu:4
#SBATCH --time=8:00:00
#SBATCH --output=/fsx/scratch/logs/groot-multi-%j.out

export DATASET_PATH="/fsx/datasets/groot/aloha"
export OUTPUT_DIR="/fsx/checkpoints/vla/groot-n1.7-aloha-multi"
export MAX_STEPS=10000
export BATCH_SIZE=64

CONTAINER_IMAGE="/fsx/scratch/nvidia+gr00t+gr00t-core+1.7.0.sqsh"

# 멀티노드 분산 학습
srun --container-image=$CONTAINER_IMAGE \
     --container-mounts=/fsx:/fsx \
     torchrun --nproc_per_node=4 \
       --nnodes=2 \
       --node_rank=$SLURM_NODEID \
       --master_addr=$(scontrol show hostnames $SLURM_JOB_NODELIST | head -n1) \
       --master_port=29500 \
       /fsx/scratch/train_groot.py
EOF
```

**멀티노드 파라미터:**

| 파라미터 | 설명 |
|---------|------|
| `--nnodes` | 총 노드 수 |
| `--node_rank` | 현재 노드 순서 (0부터 시작) |
| `--master_addr` | Master 노드 IP |
| `--master_port` | Master 노드 포트 |

## 4.7 체크포인트 확인

학습이 완료되면 체크포인트가 FSx에 저장되고, S3로 자동 익스포트됩니다:

```bash
# FSx에서 확인
ls -la /fsx/checkpoints/vla/groot-n1.7-aloha/

# 체크포인트 내용
ls /fsx/checkpoints/vla/groot-n1.7-aloha/checkpoint-*/

# 파일 크기 확인
du -sh /fsx/checkpoints/vla/groot-n1.7-aloha/*

# S3에 동기화된 체크포인트 확인 (수분 후)
aws s3 ls s3://${BUCKET}/checkpoints/vla/groot-n1.7-aloha/ --region us-east-1
```

정상적으로 완료된 체크포인트 구조:

## 4.8 트러블슈팅

| 증상 | 원인 | 해결 방법 |
|------|------|----------|
| Job이 PD(Pending)에 멈춤 | GPU 용량 부족 | 잠시 대기하거나 다른 시간에 재시도 |
| Container image not found | Enroot 이미지 생성 미완료 | `enroot import` 재실행 후 확인 |
| CUDA Out of Memory | 배치 크기 너무 큼 | `BATCH_SIZE` 줄여서 재시도 (예: 16 또는 8) |
| 로그 파일이 비어있음 | 컨테이너 실행 실패 | `scontrol show job <JOB_ID>` 에러 메시지 확인 |
| 노드 간 연결 실패 | 멀티노드 설정 오류 | `master_addr` IP 올바른지 확인 |
